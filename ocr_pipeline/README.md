# OCR Document Processing Pipeline

> **Stack:** FastAPI · Tesseract · PyMuPDF · OpenCV · OpenAI GPT-4o · Redis · Karate · Gatling  
> **Version:** 1.0.0

A modular, testable document processing pipeline that:
1. **Accepts** scanned PDF uploads (up to 20 MB)
2. **Extracts** text via Tesseract OCR (with OpenCV pre-processing)
3. **Classifies** documents using a rule-based engine + OpenAI GPT-4o
4. **Stores** results in Redis (or an in-memory fallback for local dev)

## Quick Start

### 1. Configure environment

```bash
cp .env.example .env
# Edit .env — set OPENAI_API_KEY or leave MOCK_LLM=true for local testing
```

### 2. Install Python dependencies

```bash
pip install -r requirements.txt
```

### 3. Start the API server

```bash
uvicorn main:app --reload
# → http://localhost:8000
# → Swagger UI: http://localhost:8000/docs
```

### 4. Generate test data for Karate

```bash
python generate_testdata.py
```

### 5. Run Karate functional tests

```bash
cd tests/karate
mvn test
```

### 6. Run Gatling performance tests

```bash
cd tests/karate
mvn gatling:test
# Report → target/gatling/results/<timestamp>/index.html
```

## API Reference

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/v1/upload` | Upload PDF → returns `job_id` (status: QUEUED) |
| `POST` | `/api/v1/process/{job_id}` | OCR + post-process → status: PROCESSED |
| `POST` | `/api/v1/classify/{job_id}` | Rule-based + LLM classify → status: CLASSIFIED |
| `GET`  | `/api/v1/result/{job_id}` | Full pipeline result |
| `GET`  | `/health` | Health check |

Full interactive docs: `http://localhost:8000/docs`

## Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `OPENAI_API_KEY` | *(empty)* | OpenAI API key — required for real LLM calls |
| `OPENAI_MODEL` | `gpt-4o` | Model identifier |
| `REDIS_URL` | `memory://` | Redis URL or `memory://` for in-process store |
| `MOCK_LLM` | `false` | `true` → skip OpenAI, return intermediate label |
| `MAX_FILE_SIZE_MB` | `20` | Upload size limit |
| `OCR_LANG` | `eng` | Tesseract language code |
| `RESULT_TTL_SECONDS` | `86400` | Redis key TTL (24 h) |

## Security & Vulnerability Scanning

Security scans are integrated at both the CI/CD level and the local development level using **Trivy**.

### CI/CD Pipeline Integration
Every push and pull request initiates the **Security Scans** job in GitHub Actions. This job:
1. Performs a **Filesystem and Configuration Scan** (`trivy fs`) on the workspace to detect vulnerabilities in dependencies (`requirements.txt`), leaked secrets, and Infrastructure-as-Code misconfigurations (Dockerfiles, Docker Compose, etc.).
2. Builds the Docker image and performs an **Image Scan** (`trivy image`) to detect vulnerabilities in operating system packages.
3. Fails the build (`exit-code: 1`) if any **HIGH** or **CRITICAL** vulnerabilities are found.

### Local Security Scanning
You can run the same security scans locally without needing to install Trivy on your system, by running Trivy inside a Docker container:

#### Linux, macOS, Git Bash, or WSL:
```bash
./scan-security.sh
```

#### Windows PowerShell:
```powershell
.\scan-security.ps1
```

*(Note: Ensure Docker Desktop is running before launching the scans. If you want to include local Docker image scans, first build the image: `docker build -t ocr-pipeline:latest .`)*
